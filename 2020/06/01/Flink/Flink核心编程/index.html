Flink核心编程

基于Flink1.13。

pom依赖

Flink不同版本pom文件依赖差别较大，这里仅适用于Flink1.13相关编程。

流程规范

一个基本的Flink程序处理任务的流程如下：

  1. 创建执行环境（Environment）
  2. 加载数据源（Source）
  3. 数据转换操作（Transformations）
  4. 数据输出（Sink）
  5. 触发程序执行（execute）

时间语义

时间（Time）可以分为两类：

  • 事件时间（Event Time）：数据产生的时间。
  • 处理时间（Processing Time）：数据被真正处理的时间。

在Flink的流式计算中，绝大部分的业务都会使用Event Time。一般只有在Event Time无法使用时，才会被迫使用Ingestion Time和Processing Time。
注意：Flink1.12之后默认的时间语义就是TimeCharacteristic.EventTime。

水位线

在Flink中，用来衡量事件时间（Event Time）进展的标记被称为水位线（Watermark）。
Watermark可以看作一条特殊的数据记录，它是插入到数据流中的一个标记点，主要内容就是一个时间戳，用来指示当前的事件事件。Watermark的基于数据的时间戳生成的，必须单调递增，可以通过设置延迟，来保证正确处理乱序数据。

水位线生成策略

Flink的DataStream API中提供了一个单独用于生成Watermark的方法：

Watermark Strategy包含两部分：
时间戳分配器（TimestampAssigner）：主要负责从流中数据元素的某个字段中提取时间戳，并分配给元素。时间戳的分配是生成水位线的基础。
水位线生成器（WatermarkGenerator）：主要负责按照既定的方式，基于时间戳生成水位线。在WatermarkGenerator接口中，主要有两个方法：onEvent()和 onPeriodicEmit()。其中onEvent方法为每个事件（数据）到来都会调用的方法，它的参数有当前事件、时间戳，以及允许发出水位线的一个 WatermarkOutput，可以基于事件做各种操作。onPeriodicEmit为周期性调用的方法，可以由 WatermarkOutput 发出水位线。周期时间为处理时间，可以调用环境配置的 setAutoWatermarkInterval()方法来设置，默认为200ms。

内置WatermarkGenerator

Flink提供了两个内置的水位线生成器，可以通过调用Watermark Strategy的静态辅助方法来创建。他们都是周期性生成水位线的，分别对应着处理有序流和乱序流的场景。

有序流

主要特点是时间戳单调递增（Monotonously Increasing Timestamps），永远不会出现迟到数据的情况。这是周期性生成水位线的最简单的场景，直接拿当前的最大时间戳作为水位线就可以了。

乱序流

由于乱序流中需要等待延迟数据，所以必须设置一个固定的延迟时间（Fixed Amount of Lateness）。

自定义WatermarkGenerator

一般来说，Flink内置的WatermarkGenerator就可以满足应用需求了。在 WatermarkStrategy 中，时间戳分配器 TimestampAssigner 都是大同小异的，指定字段提取时间戳就可以了；而不同策略的关键就在于 WatermarkGenerator 的实现。
整体说来，Flink有两种不同的生成水位线的方式：一种是周期性的（Periodic），另一种是断点式的（Punctuated）。

  • 周期性水位线生成器（Periodic Generator）：周期性生成器一般是通过 onEvent()观察判断输入的事件，而在 onPeriodicEmit()里发出水位线。
  • 断点式水位线生成器（Punctuated Generator）：断点式生成器会不停地检测 onEvent()中的事件，当发现带有水位线信息的特殊事件时，就立即发出水位线。一般来说，断点式生成器不会通过onPeriodicEmit()发出水位线。

窗口

在流处理中，一个很重要也很常见的操作就是窗口（Window）。所谓的窗口，一般就是划定的一段时间范围，也就是时间窗，对在这个范围内的数据进行处理，就是所谓的窗口计算。

类型

窗口可以分为两类：

  • 计算窗口（CountWindow）：按照指定的数据条数生成一个Window，与时间无关
  • 时间窗口（TimeWindow）：按照时间生成Window

对于时间窗口，可以根据窗口实现原理的不同分成三类：滚动窗口（Tumbling Window）、滑动窗口（Slibing Window）和会话窗口（Session Window）。

窗口类型 特点 适用场景
滚动窗口 时间对齐，窗口长度固定，没有重叠。 适合做每个时间段的聚合统计。
滑动窗口 时间对齐，窗口长度固定，可以有重叠。 对最近一个时间段内的统计。
会话窗口 时间无对齐。 更加精确的刻画用户行为，如刷抖音。

按键分区/非按键分区窗口（Keyed Windows & Non-Keyed Windows）

在定义窗口操作之前，首先需要确定到底是基于按键分区的数据流KeyedSteam还是直接在没有按键分区的数据流DataStream上开窗。

按键分区窗口（通用）

经过按键分区 keyBy()操作后，数据流会按照 key 被分为多条逻辑流（logical streams），这就是 KeyedStream。基于 KeyedStream 进行窗口操作时, 窗口计算会在多个并行子任务上同时执行。相同 key 的数据会被发送到同一个并行子任务，而窗口操作会基于每个 key 进行单独的处理。所以可以认为每个 key 上都定义了一组窗口，各自独立地进行统计计算。

非按键分区窗口（不推荐）

如果没有进行 keyBy()，那么原始的 DataStream 就不会分成多条逻辑流。这时窗口逻辑只能在一个任务（task）上执行，就相当于并行度变成了 1。所以在实际应用中一般不推荐使用这种方式。
注意：对于非按键分区的窗口操作，手动调大窗口算子的并行度也是无效的，windowAll 本身就是一个非并行的操作。

窗口分配器

窗口分配器（Window Assigners）其实就是在指定窗口的类型。
窗口分配器最通用的定义方式，就是调用 window()方法。这个方法需要传入一个WindowAssigner 作为参数，返回WindowedStream。如果是非按键分区窗口，那么直接调用windowAll()方法，同样传入一个 WindowAssigner，返回的是 AllWindowedStream。

窗口函数

经窗口分配器处理之后，数据可以分配到对应的窗口中，而数据流经过转换得到的数据类型是 WindowedStream。这个类型并不是 DataStream，所以并不能直接进行其他转换，而必须进一步调用窗口函数（Window Functions），对收集到的数据进行处理计算之后才能最终再次得到 DataStream。

窗口函数定义了要对窗口中收集的数据做的计算操作。主要可以分为两类：

  • 增量聚合函数（incremental aggregation functions））：每条数据到来就计算，保持一个简单的状态。典型的增量聚合函数有ReduceFunction、AggregateFunction。
  • 全窗口函数（full window functions）：先把窗口所有数据收集起来，等到计算的时候会遍历所有数据。典型的全窗口函数有ProcessWindowFunction。

ReduceFunction

AggregateFunction

ReduceFunction 可以解决大多数归约聚合的问题，但是这个接口有一个限制，就是聚合状态的类型、输出结果的类型都必须和输入数据类型一样。为了更加灵活地处理窗口计算，Flink的Window API提供了更加一般化的aggregate()方法。直接基于 WindowedStream 调用 aggregate()方法，就可以定义更加灵活的窗口聚合操作。
AggregateFunction 可以看作是 ReduceFunction 的通用版本，这里有三种类型：
输入类型（IN）：输入流中元素的数据类型
累加器类型（ACC）：进行聚合的中间状态类型
输出类型（OUT）：最终计算结果的数据类型

ProcessWindowFunction

在 Flink 中，全窗口函数有两种：WindowFunction 和 ProcessWindowFunction。WindowFunction 字面上就是“窗口函数”，它其实是老版本的通用窗口函数接口。WindowFunction 能提供的上下文信息较少，也没有更高级的功能。事实上，它的作用可以被 ProcessWindowFunction 全覆盖，所以之后可能会逐渐弃用。一般在实际应用，直接使用 ProcessWindowFunction 就可以了。

增量聚合函数和全窗口聚合函数结合

我们之前在调用 WindowedStream 的 reduce()和 aggregate()方法时，只是简单地直接传入了一个 ReduceFunction 或 AggregateFunction 进行增量聚合。除此之外，其实还可以传入第二个参数：一个全窗口函数，可以是 WindowFunction 或者 ProcessWindowFunction。

这样调用的处理机制是：基于第一个参数（增量聚合函数）来处理窗口数据，每来一个数据就做一次聚合；等到窗口需要触发计算时，则调用第二个参数（全窗口函数）的处理逻辑输出结果。需要注意的是，这里的全窗口函数就不再缓存所有数据了，而是直接将增量聚合函数的结果拿来当作了 Iterable 类型的输入。一般情况下，这时的可迭代集合中就只有一个元素了。

状态

在Flink中，算子任务可以分为无状态和有状态两种情况。
无状态算子：无状态的算子任务只需要观察每个独立事件，根据当前输入的数据直接转换输出结果，基本转换算子如map、filter、flatMap都属于无状态的算子。
有状态算子：有状态的算子除当前数据外，还需要一些其他数据来得到计算结果。这里的其他数据，就是所谓的状态。最常见的就是之前到达的数据或者由之前数据计算出的某个结果。比如求和计算时，需要保存之前所有数据的和，这就是状态。

状态分类

Flink的状态有两种：

  • 托管状态（Managed State）：由Flink统一管理。
  • 原始状态（Raw State）：全部需要自定义。

托管状态又可以分为两类：算子状态和键控状态。

算子状态（Operator State）

算子状态可以用在所有算子上，使用的时候和一个本地变量没什么区别。
算子状态支持的结构类型有：

  • 列表状态（List State）：将需要保存的数据，以列表的形式保存。
  • 联合列表状态（Union List State）
  • 广播状态（BroadcastState）

键控状态（Keyed State）

键控状态是根据输入流中定义的Key来维护和访问的，所以只能定义在KeyedStream中。
键控状态支持的结构类型有：

  • 值状态（Value State）：状态中只保存一个值。这个值可以是任意数据类型。
  • 列表状态（List State）：将需要保存的数据，以列表的形式保存。
  • 映射状态（Map State）：将需要保存的数据，以键值对的形式保存。
  • 归约状态（Reducing State）：类似于值状态，将归约聚合之后的值作为状态保存下来。
  • 聚合状态（Aggregating State）：与归约状态类似，将聚合之后的值作为状态保存下来，但是更加灵活。

状态后端（state backend）

Flink对状态进行持久化的方式就是将当前所有分布式状态进行快照保存，写入一个检查点（Checkpoint）或者保存点（Savapoint）保存到外部存储系统中。
在Flink中，状态的存储、访问以及维护，都是由一个可插拔的组件决定的，这个组件就叫做状态后端。Flink提供了两种不同类型的状态后端：

  • 哈希表状态后端（HashMapStateBackend）：状态存放在内存中。
  • 内嵌 RocksDB 状态后端（EmbeddedRocksDBStateBackend）：状态持久化到本地硬盘。

注意：如果想在 IDE 中使用 EmbeddedRocksDBStateBackend，需要添加依赖。
