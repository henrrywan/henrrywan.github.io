Flink核心编程

一个基本的Flink程序处理任务的流程如下：

  1. 获取执行环境（Environment）
  2. 加载数据源（Source）
  3. 数据转换操作（Transformations，业务处理）
  4. 数据输出（Sink）
  5. 执行任务（execute）

用户自定义函数处理业务

Flink 的 DataStream API 编程风格其实是一致的：基本上都是基于 DataStream 调用一个方法，表示要做一个转换操作；方法需要传入一个参数，这个参数都是需要实现一个接口。
这些接口有一个共同特点：全部都以算子操作名称 + Function 命名，例如源算子需要实现 SourceFunction 接口，map 算子需要实现 MapFunction 接口，reduce()算子需要实现
ReduceFunction 接口。我们不仅可以通过自定义函数类或者匿名类来实现接口，也可以直接传入 Lambda 表达式。这就是所谓的用户自定义函数（user-defined function，UDF）。

Function Classes

对于大部分操作而言，都需要传入一个用户自定义函数（UDF），实现相关操作的接口，
来完成处理逻辑的定义。Flink 暴露了所有 UDF 函数的接口，具体实现方式为接口或者抽象类，例如 MapFunction、FilterFunction、ReduceFunction 等。
所以最简单直接的方式，就是自定义一个函数类，实现对应的接口。

Rich Function Classes

“富函数类”也是 DataStream API 提供的一个函数类的接口，所有的 Flink 函数类都有其Rich 版本。富函数类一般是以抽象类的形式出现的。例如：RichMapFunction、RichFilterFunction、RichReduceFunction 等。与常规函数类的不同主要在于，富函数类可以获取运行环境的上下文，并拥有一些生命周期方法，所以可以实现更复杂的功能。

Flink CEP

所谓CEP，其实就是复杂事件处理（Complex Event Processing）的缩写。Flink CEP是Flink实现的一个用于复杂事件处理的库。
复杂事件处理是指在事件流里检测到特定的事件组合并进行处理。比如连续登录失败、订单支付超时等。
CEP的处理流程可以分为三个步骤：

  1. 定义一个匹配规则
  2. 将匹配规则应用到事件流上，检测满足规则的复杂事件
  3. 对检测到的复杂事件进行处理，得到想要的结果进行输出。

Pattern

CEP的第一步所定义的匹配规则，我们称之为模式（Pattern）。模式的定义主要包括两部分：

  1. 每个简单事件的特征
  2. 简单事件之间的组合关系
