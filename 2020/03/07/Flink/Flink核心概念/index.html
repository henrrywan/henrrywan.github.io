Flink核心概念

前言

在这一章节我们主要学习Flink中相关的核心概念。具体包括：

  • Stream（流）
  • State（状态）
  • Time（时间）
  • Tasks and Operator Chains（任务和算子链）
  • Job Managers, Task Managers, Clients
  • Task Slots and Resources（任务槽和资源）

核心概念

Stream

在Flink的开篇中，我们已经详细了介绍了Stream的概念，这里用四句话简单概括：

  • 任何类型的数据都可以形成一种事件流。
  • 流分为有界流和无界流。
  • 有界流定义了流的开始，也定义了流的结束（有界流处理通常被称为批处理）。
  • 无界流定义了流的开始，但是没有定义流的结束。

State

所谓的状态，是指在进行流式计算过程中的信息。一般用于容错恢复和持久化，流式计算在本质上是增量计算，也就是说需要不断查询过去的状态。
在Flink中，每一个具有一定复杂度的流处理应用都是有状态的。只有在每一个单独的事件上进行转换操作的应用才不需要状态。任何运行基本业务逻辑的流处理应用都需要在一定时间内存储所接收的事件或中间结果，以供后续的某个时间点进行访问并进行后续处理。
State在Flink中有十分重要的作用，例如为了确保Exactly-once语义需要将数据写入到State中。

Time

在Flink中，支持三种不同的Time概念。分别是Event Time、Ingestion Time以及Processing Time。Time是我们在进行Flink程序开发时判断业务状态是否滞后和延迟的重要依据。

从这张图片我们可以清楚的看到三种不同的Time之间的关系。这里简单说明：

  • Event Time：事件时间。是指事件发生的时间，一般就是数据本身携带的时间。这个时间通常在事件到达Flink之前就已经确定了，并且可以从每个事件中获取到事件时间戳。在Event Time中，时间取决于数据，跟其它没什么关系。Event Time程序必须指定如何生成Event Time水印，这是表示Event Time进度的机制。
  • Ingestion Time：摄取时间。是指事件进入Flink的时间，在源操作处，每个事件将源的当前时间作为时间戳，并且基于时间的操作（如时间窗口）会利用到这个时间戳。与Event Time相比，Ingestion Time程序无法处理任何无序事件或延迟数据，但程序不需要指定如何生成水印。
  • Processing Time：处理时间。是指事件被处理时机器的系统时间。当Flink流程序在Processing Time上运行时，所有基于时间的操作（如时间窗口）都将使用当时机器的系统时间。例如，如果应用程序在上午9:15 开始运行，则第一个每小时 Processing Time 窗口将包括在上午 9:15 到上午 10:00 之间处理的事件，下一个窗口将包括在上午 10:00 到 11:00 之间处理的事件。相较于其他Time概念，Processing Time是最简单的时间概念，它不需要流和机器之间的协调，它提供了最好的性能和最低的延迟。但是，在分布式环境下，Processing Time不能提供确定性，因为它容易受到事件到达系统的速度（例如从消息队列中）、事件在系统内操作流动的速度以及中断的影响。

Tasks and Operator Chains

在分布式运行环境中，Flink提出了算子链的概念。Flink将多个算子放在一个Task中，由同一个线程执行（每个任务由一个线程执行）。

图中的示例数据流由五个子任务执行，因此具有五个并行线程。
将算子链接到任务中的一个好处就是可以减少线程之间的切换、消息序列化反序列化、数据在缓冲区的交换以及减少延迟的同时提高整体的吞吐量。

Job Managers, Task Managers, Clients

在实际生产环境中，Flink都是以集群的方式运行，运行的过程中包含两种类型的进程：

  • Job Managers：也称为Masters，集群管理者角色，负责任务调度，协调checkpoints、协调故障恢复等。在集群中，总是至少有一个Job Manager，高可用的情况下则有多个Job Manager，其中一个是leader，其它的处于standby。
  • Task Managers：也称为Workers，负责任务执行。同时Task Manager也是所在节点的管理者，它负责把该节点上的信息比如内存、磁盘资源、任务运行情况等向Job Manager汇报。
  • Client：用户在提交编写好的Flink工程时，会先创建一个客户端再进行提交，这个客户端就是Client。Client会根据用户传入的参数选择使用yarn per job模式或者stand-alone模式或者yarn-session模式来将Flink程序提交到集群。

Task Slots and Resources

在Flink集群中，一个TaskManager就是一个JVM进程，并且会用独立的线程来执行Task。为了控制一个TaskManager能接收多少个Task，Flink提出了Task Slot的概念。我们可以把Task Slot理解为TaskManager的计算资源的子集。假设一个TaskManger有三个slot，那么该TaskManager的计算资源就会被平均分成3份，不同的Task在不同的slot中执行，避免资源竞争。需要注意的是，slot仅仅用来做内存的隔离，对CPU不起作用。运行在同一个JVM中的Task可以共享TCP连接，减少网络传输，在一定程度上，能够提高程序的运行效率，降低资源消耗。

默认情况下，Flink允许子任务共享Slot，即使他们是不同任务的子任务，只要他们来自同一个作业。

总结

这一章节我们主要学习了Flink的核心概念，对相关概念有一个初步的了解，在后续的学习中进一步深入理解。
